¿Qué es la Sobrecarga de Operadores?

Dándole nuevo significado a símbolos existentes
Definición formal

La sobrecarga de operadores (operator overloading) es la capacidad de asignarle un comportamiento personalizado a operadores existentes (como +, -, *, <<) cuando se usan con tipos de datos definidos por el usuario (clases y structs).

Es una forma de polimorfismo en tiempo de compilación: el compilador decide qué función invocar en función de los tipos de los operandos.

Idea central

El operador + ya sabe sumar int, double y float. Con sobrecarga, tú le enseñas cómo sumar tus propios objetos.

Metáfora — El control universal
📺

Un mismo botón, distintas acciones según el dispositivo

Imagina un control remoto universal. El botón "▶ Play" siempre significa "reproducir", pero lo que hace depende de a qué aparato esté apuntando: en el DVD reproduce una película, en el estéreo reproduce música, en el Blu-ray reproduce en alta definición.

El operador + es ese botón: siempre significa "combinar", pero tú defines qué hace cuando los operandos son tus objetos.

Metáfora alternativa — El idioma
🌍

Una palabra, distintos significados según el contexto

En español, el símbolo "+" en matemáticas significa suma; en música podría indicar una nota sostenida; en medicina indica positivo. La notación es la misma, el significado depende del dominio.

Con sobrecarga le estás diciendo al compilador: "cuando veas obj1 + obj2 y ambos son de mi clase, aquí tienes las instrucciones exactas de qué hacer".

Antes vs después de la sobrecarga
Sin sobrecarga — verbose y confuso
// Tienes que llamar métodos explícitos
Rect resultado;
resultado.setX( r1.getX() + r2.getX() );
resultado.setY( r1.getY() + r2.getY() );
Con sobrecarga — intuitivo y legible
// Igual que con int o double
Rect resultado = r1 + r2;

Ambos producen el mismo resultado, pero la versión con sobrecarga es más legible, natural y segura porque la lógica de combinación vive dentro de la clase, no dispersa en el cliente.

¿Por Qué Usar Sobrecarga de Operadores?

Motivación con el escenario de coordenadas
El escenario del material de clase

Supón que tienes dos clases para representar puntos en un plano:

Coordenada Rectangular (Rect)

Formato: (x, y)

Ejemplo: (3, 4)

La suma y resta son simples: se suman las partes correspondientes.

suma.x = r1.x + r2.x;
suma.y = r1.y + r2.y;

Coordenada Polar (Pol)

Formato: (r, θ)

Ejemplo: (4, 30°)

La multiplicación y división son simples en forma polar: se multiplican módulos y suman ángulos.

prod.r = p1.r * p2.r;
prod.ang = p1.ang + p2.ang;
Beneficios de la sobrecarga

Legibilidad

El código cliente lee como matemática, no como código de máquina.

Encapsulamiento

La lógica de operación vive dentro de la clase, no regada en el main.

Interoperabilidad

Puedes definir cómo se combina un Rect con un Pol de forma transparente.

Código completo — Rect con suma sobrecargada
rect.h// Clase Rect con sobrecarga de +
class Rect {
public:
    double x, y;

    Rect(double x = 0, double y = 0) : x(x), y(y) {}

    // Sobrecarga de +
    Rect operator+(const Rect& otro) const {
        return Rect(x + otro.x, y + otro.y);
    }

    // Sobrecarga de -
    Rect operator-(const Rect& otro) const {
        return Rect(x - otro.x, y - otro.y);
    }
};

main.cppRect r1(1, 2), r2(3, 4);
Rect suma = r1 + r2;    // compila: llama a r1.operator+(r2)
// suma.x = 4,  suma.y = 6
Lo que hace el compilador

Cuando ve r1 + r2, el compilador lo transforma internamente en r1.operator+(r2). Sin la sobrecarga sería un error de compilación porque no sabe cómo combinar dos objetos de tipo Rect.

Sintaxis de la Sobrecarga

La palabra clave operator
Forma general
// Como función miembro
TipoRetorno operatorsímbolo(Parámetros) const;

// Como función no-miembro (friend)
friend TipoRetorno operatorsímbolo(Param1, Param2);

La palabra reservada operator seguida del símbolo indica al compilador que esta función define el comportamiento de ese operador para los tipos de los parámetros.

Función miembro vs función no-miembro

Función miembro

  • El objeto izquierdo es *this
  • Solo necesita un parámetro (operando derecho)
  • Tiene acceso directo a datos privados
Complejo operator+(const Complejo& c) {
    return Complejo(real+c.real,
                   imag+c.imag);
}

Función no-miembro (friend)

  • Necesita dos parámetros
  • Necesaria cuando el operando izquierdo no es de tu clase
  • Obligatoria para << y >>
friend ostream& operator<<(
    ostream& os,
    const Complejo& c);
Ejemplo completo — Números complejos
complejo.h / complejo.cppclass Complejo {
private:
    double real, imag;
public:
    Complejo(double r = 0, double i = 0) : real(r), imag(i) {}

    Complejo operator+(const Complejo& c) const {
        return Complejo(real + c.real, imag + c.imag);
    }

    friend ostream& operator<<(ostream& os, const Complejo& c) {
        os << c.real << " + " << c.imag << "i";
        return os;  // regresa os para poder encadenar: cout << a << b
    }
};

// Uso:
Complejo c1(3, 5), c2(2, 4);
Complejo c3 = c1 + c2;
cout << c3;  // Output: 5 + 9i

Reglas y Restricciones

Lo que puedes y no puedes hacer — Deitel Ch. 10 + GeeksforGeeks
Reglas doradas
  1. Al menos un operando debe ser un tipo definido por el usuario. No puedes sobrecargar int + int ni cambiar el comportamiento de los tipos primitivos.
  2. No puedes cambiar la precedencia ni la asociatividad. Si * tiene mayor precedencia que + para ints, la sigue teniendo para tus clases.
  3. No puedes cambiar la aridad (número de operandos). Un operador binario siempre necesita dos operandos.
  4. No puedes crear operadores nuevos (ej: ** para potencia). Solo puedes reutilizar los símbolos existentes.
  5. Algunos operadores deben ser funciones miembro. Ver tabla abajo.
Operadores que NO se pueden sobrecargar
Operador Nombre ¿Por qué no?
sizeofTamaño Se evalúa en compilación; afectar el layout de memoria causaría caos
typeidIdentificación de tipo Debe identificar tipos únicos; sobrecargar rompería RTTI
::Resolución de alcance Trabaja sobre nombres, no valores; resuelto en compilación
. y .*Acceso a miembro El compilador los usa internamente; sobrecargar rompería el acceso a objetos
?:Ternario condicional Evalúa solo una rama; funciones no pueden replicar la evaluación diferida
Operadores que DEBEN ser funciones miembro
= [] () ->

Estos cuatro siempre deben declararse como miembros de la clase porque actúan directamente sobre el objeto y el estándar C++ así lo exige.

Error común

Si intentas declarar operator= como función no-miembro, el compilador lo rechazará con un error como: "operator= must be a non-static member function".

Operadores que DEBEN ser no-miembro (friend)
<< >>

El operador de salida << y el de entrada >> siempre deben ser funciones no-miembro, porque el operando izquierdo es ostream o istream, que no son tu clase.

Se declaran como friend dentro de la clase para que puedan acceder a los datos privados.

Sobrecarga de Operadores Binarios

+, −, *, /, ==, !=, <, > y más
Ejemplo 1 — Casilla de juego (contexto de la Tarea 3)

En Serpientes y Escaleras, una casilla normal avanza el número de pasos que indica el dado. Una casilla serpiente retrocede según su penalización. Con sobrecarga del +, el código cliente no necesita saber qué tipo de casilla está usando:

casilla.hclass Casilla {
protected:
    int posicion;
public:
    Casilla(int pos) : posicion(pos) {}
    int getPosicion() const { return posicion; }

    // Operador + base: avanza normalmente
    virtual int operator+(int pasos) const {
        return posicion + pasos;
    }
};

class Serpiente : public Casilla {
private:
    int penalty;   // siempre positivo
public:
    Serpiente(int pos, int p) : Casilla(pos), penalty(p) {}

    // "+" con serpiente SIEMPRE resta el valor absoluto
    int operator+(int pasos) const override {
        return posicion - abs(penalty);
    }
};

main.cpp — uso transparenteCasilla* c = new Serpiente(5, 4);
int nueva_pos = *c + 4;   // resultado: 5 - 4 = 1  ✓
cout << "Nueva posición: " << nueva_pos;
Explicación de la tarea

La especificación dice: si la casilla origen es 5 y caes en una serpiente con penalty 4, entonces 5 + 4 = 1 (se resta el absoluto). La sobrecarga permite que el símbolo + haga la resta internamente, y el código del juego queda limpio y uniforme.

Ejemplo 2 — Clase Array con [] (Deitel Ch.10)

Deitel usa una clase Array para mostrar cómo sobrecargar [] con comprobación de rangos, algo que el array nativo de C++ no hace:

class Array {
private:
    int* datos;
    int tam;
public:
    Array(int t) : tam(t), datos(new int[t]()) {}

    // [] con comprobación de rango
    int& operator[](int idx) {
        if (idx < 0 || idx >= tam)
            throw out_of_range("Índice fuera de rango");
        return datos[idx];
    }

    // == para comparar dos arrays
    bool operator==(const Array& otro) const {
        if (tam != otro.tam) return false;
        for (int i = 0; i < tam; i++)
            if (datos[i] != otro.datos[i]) return false;
        return true;
    }
};

OperadorFirma típicaDevuelve
+ - * / T operator+(const T& o) const Nuevo objeto T
+= -= T& operator+=(const T& o) Referencia a *this
== != bool operator==(const T& o) const bool
< > <= >= bool operator<(const T& o) const bool
[] T& operator[](int idx) Referencia al elemento

Sobrecarga del Operador << (Stream)

Impresión personalizada con cout — también aplica para >>
¿Por qué << debe ser no-miembro?
🖨️

Metáfora — La impresora y el documento

El operador << es como una impresora: su operando izquierdo es siempre la impresora (cout, un objeto de tipo ostream), y el derecho es el documento (tu objeto).

Como la impresora no es tu clase, no puedes añadirle un método miembro. En cambio, defines una función externa que sabe cómo imprimir tu objeto.

Por qué no funciona como miembro// SI fuera miembro de Turno, el uso sería al revés:
turno.operator<<(cout);   // ← raro y no compila con cout << turno

// Como friend no-miembro:
cout << turno;             // ← natural y correcto
// que el compilador traduce a:
operator<<(cout, turno);  // ← llama a la función friend
Clase Turno — Implementación para la Tarea 3
turno.hclass Turno {
private:
    int    jugador;
    int    dado;
    int    posInicial;
    int    posFinal;
    string evento;   // "Escalera", "Serpiente", "Normal"

public:
    Turno(int jug, int dado, int posI, int posF, const string& ev)
        : jugador(jug), dado(dado), posInicial(posI),
          posFinal(posF), evento(ev) {}

    // Sobrecarga de << — DEBE ser friend no-miembro
    friend ostream& operator<<(ostream& os, const Turno& t);
};

turno.cppostream& operator<<(ostream& os, const Turno& t) {
    os << "Jugador "  << t.jugador
       << " | Dado: " << t.dado
       << " | Pos: "  << t.posInicial << " -> " << t.posFinal
       << " | "       << t.evento;
    return os;  // CRÍTICO: devolver os permite encadenar <<
}

main.cpp — usoTurno t(1, 5, 10, 15, "Normal");
cout << t << endl;
// Output: Jugador 1 | Dado: 5 | Pos: 10 -> 15 | Normal

// Encadenamiento funciona porque regresamos ostream&:
cout << t1 << "\n" << t2 << endl;
Error frecuente — no devolver ostream&

Si la función regresa void en lugar de ostream&, el encadenamiento cout << t1 << t2 no compilará. Siempre regresa la referencia al stream.

Ejemplo de Deitel — PhoneNumber

Deitel (Ch. 10) usa PhoneNumber para mostrar cómo sobrecargar << y >> para dar formato automático:

// Imprime: (800) 555-1212
friend ostream& operator<<(ostream& out, const PhoneNumber& num) {
    out << "(" << num.areaCode << ") "
        << num.exchange  << "-"
        << num.line;
    return out;
}

// Lee: "(800) 555-1212" y separa automáticamente
friend istream& operator>>(istream& in, PhoneNumber& num) {
    in.ignore();                      // ignora '('
    in >> setw(3) >> num.areaCode;   // lee 3 dígitos
    in.ignore(2);                    // ignora ') '
    in >> setw(3) >> num.exchange;
    in.ignore();                      // ignora '-'
    in >> setw(4) >> num.line;
    return in;
}

Operadores de Conversión

Convertir tu objeto a otros tipos automáticamente
Constructor de conversión

Cualquier constructor que acepta un solo argumento actúa como un conversor implícito. Cuando el compilador necesita el tipo de tu clase y tiene otro tipo disponible, llama a este constructor automáticamente.

class Punto {
public:
    int x, y;
    Punto(int i = 0, int j = 0) : x(i), y(j) {}
};

Punto p;
p = 30;   // Llama Punto(30), crea Punto(30,0) y lo asigna
           // p.x = 30,  p.y = 0
Uso de explicit

Las conversiones implícitas pueden causar bugs sutiles. Usa explicit en el constructor para forzar conversiones explícitas:

explicit Punto(int i) : x(i), y(0) {}
// Ahora  p = 30;  es error de compilación
// Solo    p = Punto(30);  funciona
Operador de conversión (cast)

Permite que tu objeto se convierta automáticamente a un tipo primitivo o a otro tipo de clase. Siempre debe ser función miembro y no tiene tipo de retorno declarado explícitamente:

class Fraccion {
private:
    int num, den;
public:
    Fraccion(int n, int d) : num(n), den(d) {}

    // Conversión implícita Fraccion -> float
    operator float() const {
        return static_cast<float>(num) / den;
    }
};

Fraccion f(2, 5);
float val = f;    // Llama a operator float() → 0.4
cout << val;      // Output: 0.4
Reto del material de clase — Rect + Pol
El reto de la presentación original

Sobrecargar + para Rect cuando el otro operando es Pol. La solución: el parámetro es de tipo Pol, se convierte a Rect, se suman las partes, y se regresa un Rect.

// En la clase Rect:
Rect operator+(const Pol& polar) const {
    Rect convertido = polar.toRect();  // conversión de Pol a Rect
    return Rect(x + convertido.x,
                y + convertido.y);
}

// Uso:
Rect r(1, 2);
Pol  p(4, 30);    // módulo 4, ángulo 30°
Rect res = r + p;  // Completamente válido con la sobrecarga

Caso real — std::string como inspiración

La clase std::string de la STL es el ejemplo más cotidiano de sobrecarga. Internamente implementa docenas de operadores sobrecargados:

string nombre = "Ana";
string apellido = "García";

string completo = nombre + " " + apellido;   // operator+
bool   igual    = nombre == apellido;         // operator==
char   inicial  = nombre[0];                  // operator[]
cout << completo;                             // operator<<

Si std::string no tuviera sobrecarga, escribirías: nombre.concat(" ").concat(apellido). La sobrecarga hace que tu código lea como el problema que resuelve, no como las instrucciones de la máquina.

Caso real — Motor de videojuego (tipo Unity/Godot)
class Vector2 {
public:
    float x, y;

    Vector2 operator+(const Vector2& v)  const { return {x+v.x, y+v.y}; }
    Vector2 operator*(float t)            const { return {x*t,   y*t  }; }
    Vector2& operator+=(const Vector2& v) { x+=v.x; y+=v.y; return *this; }
    bool     operator==(const Vector2& v) const { return x==v.x && y==v.y; }
};

// En el loop del juego — se lee como física, no como código
Vector2 posicion(0, 0), velocidad(2, 0);
float dt = 0.016f;  // ~60 FPS

posicion += velocidad * dt;   // mueve el personaje cada frame
